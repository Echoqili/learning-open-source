# Superpowers + OpenSpec 总控串联 Skill

## 触发条件

当用户表达以下任一意图时激活本 Skill：

- “用 Superpowers + OpenSpec 跑一遍”
- “启动总控 Skill”
- “Brainstorm → 生成规范 → TDD 开发”
- 输入原始业务需求并准备进入完整开发流水线

## 核心原则

1. **需求锁定**：OpenSpec 生成的三份文档是后续编码的“宪法”，任何阶段都不得新增需求或改方案。
2. **阶段隔离**：
   - Brainstorm 只产出内存草稿，**不写任何本地文件**。
   - OpenSpec 只写 `spec.md` / `plan.md` / `task.md` 三份文档，**不写代码**。
   - TDD 编码只新增代码与测试，**禁止修改上述三份文档**。
3. **测试先行**：每个任务必须先写测试，再写实现，测试失败时只改代码不改文档。

## 执行总序

```text
原始需求输入
    │
    ▼
阶段一：Brainstorm（仅内存，无文件输出）
    │
    ▼
阶段二：OpenSpec 生成规范（spec.md / plan.md / task.md）
    │
    ▼
阶段三：受控 TDD 编码（仅新增代码/测试）
    │
    ▼
阶段四：校验与报告
```

---

## 阶段一：Brainstorm（无文件输出）

以 `superpowers.brainstorm` 角色执行：

1. 接收用户原始需求，确认边界与隐式约束（主动询问不清晰点）。
2. 完成以下思考并整理为 **结构化 BrainstormDraft**（仅保存在当前对话上下文/内存变量中）：
   - 业务目标与成功标准
   - 用户场景与关键路径
   - 可行方案发散（至少 2 个方案）
   - 推荐方案及理由
   - 主要风险、依赖、降级方案
   - 需要进一步确认的问题（如有）
3. **硬性约束**：本阶段不得写入本地任何文件（包括但不限于 `.md`、`.txt`、`.json`、临时文件）。所有输出必须是可复用的结构化草稿文本，供下一阶段读取。

BrainstormDraft 输出格式示例：

```markdown
# BrainstormDraft

## 目标
...
## 用户场景
...
## 候选方案
- 方案 A：...
- 方案 B：...
## 推荐方案
...
## 风险与依赖
...
## 待确认问题
...
```

---

## 阶段二：OpenSpec 生成规范文档

以 `speckit(OpenSpec)` 工作流角色执行，将 BrainstormDraft 作为唯一输入：

1. 在项目根目录生成三份 Markdown 文档：
   - `spec.md`：需求规格（proposal 提案层）
   - `plan.md`：开发方案（design 设计层）
   - `task.md`：细分任务（tasks 任务层）
2. 严格按下方模板填充，**不写代码、不写测试、不创建实现文件**。
3. 生成后向用户展示三份文档的摘要与文件路径。

### 2.1 spec.md 模板（需求规格）

```markdown
# Proposal: <功能标题>

## 1. 背景与目标
- 业务背景：...
- 功能目标：...
- 成功指标：...

## 2. 用户故事 / 验收标准
| 编号 | 用户故事 | 验收标准（可验证） |
|------|----------|-------------------|
| AC-1 | 作为...，我希望...，以便... | 当...时，期望... |

## 3. 范围
### In Scope
- ...
### Out of Scope
- ...

## 4. 功能需求
- FR-1: ...
- FR-2: ...

## 5. 非功能需求
- 性能：...
- 安全：...
- 可维护性：...

## 6. 约束与假设
- ...

## 7. 验收清单
- [ ] AC-1 已实现并通过测试
- [ ] AC-2 已实现并通过测试
- [ ] ...
```

### 2.2 plan.md 模板（开发方案）

```markdown
# Design: <功能标题>

## 1. 架构概览
- 模块关系图（文字描述或 Mermaid）
- 数据流向

## 2. 技术选型
| 层级 | 选型 | 理由 |
|------|------|------|
| 语言/框架 | ... | ... |
| 测试框架 | ... | ... |
| 构建/部署 | ... | ... |

## 3. 模块划分
| 模块 | 职责 | 关键接口/文件 |
|------|------|--------------|
| ... | ... | ... |

## 4. 数据模型 / API 契约
- 实体定义
- 请求/响应格式
- 错误码约定

## 5. 错误处理与日志
- ...

## 6. 测试策略
- 单元测试覆盖率目标
- 集成测试范围
- Mock 策略

## 7. 风险与降级方案
| 风险 | 影响 | 应对措施 |
|------|------|----------|
| ... | ... | ... |
```

### 2.3 task.md 模板（细分任务）

```markdown
# Tasks: <功能标题>

> 说明：本表由 OpenSpec 生成，编码阶段仅作为执行蓝图，**不得修改本文档内容**。

## 任务列表

| ID | 优先级 | 依赖 | 任务描述 | 验收标准 | 估算 |
|----|--------|------|----------|----------|------|
| T1 | P0 | - | 搭建项目骨架与依赖 | 项目可编译/运行 | ... |
| T2 | P0 | T1 | 编写 XX 的单元测试（Red） | 测试运行且失败 | ... |
| T3 | P0 | T2 | 实现 XX 通过测试（Green） | 测试通过 | ... |
| ... | ... | ... | ... | ... | ... |

## 依赖图
```text
T1 -> T2 -> T3 -> ...
```
```

## 阶段三：受控 TDD 编码

以 `superpowers.coding` 角色执行：

### 3.1 前置读取

1. 读取 `spec.md`、`plan.md`、`task.md` 三份文档。
2. 在内存中建立任务队列，按依赖顺序排列。

### 3.2 每个任务的 TDD 循环

对每个任务执行：

1. **Red**：先写测试用例（单元测试或集成测试），运行并确认失败。
2. **Green**：编写最少量业务代码使测试通过。
3. **Run**：运行对应测试；若失败，仅修改代码/测试，**不修改三份规范文档**。
4. **Refactor**：在测试通过前提下重构，保持行为不变。

### 3.3 硬性约束

- **仅新增**：代码文件、测试文件、测试用例、配置文件、依赖声明。
- **禁止修改**：`spec.md`、`plan.md`、`task.md` 的任何已有内容。
- **禁止新增需求**：不得实现未在 `spec.md` 验收标准中列出的功能。
- **禁止改方案**：技术选型、模块划分、接口契约必须遵循 `plan.md`。
- **禁止跳测**：每个任务至少对应一个可运行的测试用例。
- **测试框架/语言**：严格遵循 `plan.md` 中的选型。

### 3.4 完成报告

每完成一个任务，在对话中简要报告：

- 任务 ID 与描述
- 新增/修改的文件（仅代码与测试）
- 测试结果摘要（通过/失败数）

---

## 阶段四：校验与报告

编码全部完成后执行以下校验：

1. **文档完整性校验**
   - 若项目使用 git，运行 `git diff -- spec.md plan.md task.md`，确保三份文档无变更。
   - 若无 git，对比生成后的快照，确保内容一致。
2. **TDD 校验**
   - 运行完整测试套件，确保全部通过。
   - 检查每个 `task.md` 任务都有对应测试覆盖。
3. **需求对齐校验**
   - 对照 `spec.md` 的验收标准逐项勾选：
     - 已实现并通过测试 → 勾选
     - 未实现或偏离 → 标记并说明
4. 输出最终总结：
   - 三份规范文档是否未被改动（是/否）
   - 测试总数 / 通过数 / 失败数
   - 已实现验收标准清单
   - 待确认/遗留问题

---

## 反模式（必须避免）

| 反模式 | 正确做法 |
|--------|----------|
| Brainstorm 阶段写出临时方案文档 | 仅保留在内存中 |
| 编码时觉得 spec 不合理就顺手改 spec | 先暂停并向用户请示，获准后回到阶段二重新生成规范，再进入阶段三 |
| 先写实现再补测试 | 必须先写测试（Red → Green → Refactor） |
| 为了省事跳过单元测试直接集成测试 | 按 `plan.md` 测试策略执行 |
| 实现中临时加入未在 spec 中的功能 | 超出范围，必须剔除或重新走 OpenSpec 流程 |

## 一键日常口令示例

用户可以说：

> “帮我用 Superpowers + OpenSpec 流水线实现：用户登录后能看到自己的订单列表，支持按状态筛选。”

本 Skill 自动按阶段一 → 阶段二 → 阶段三 → 阶段四执行。
